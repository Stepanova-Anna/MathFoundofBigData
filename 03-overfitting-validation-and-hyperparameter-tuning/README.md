- **Почему поиск параметров велся только на `train`:**
  - GridSearchCV должен искать гиперпараметры только на обучающих данных. Validation используется только для финального сравнения tuned-моделей, а test — только для однократной финальной проверки. Это гарантирует честность оценки.
- **Почему `Pipeline` помогает избежать leakage только тогда, когда все обучаемые preprocessing steps находятся внутри CV:**
  - Pipeline гарантирует, что препроцессинг (стандартизация, кодирование) выполняется внутри каждого фолда CV, а не на всей выборке до разбиения. Если препроцессинг выполняется до CV, информация из валидации может просочиться в обучение, и метрики станут смещёнными.

## 8. Проверка понимания

1. **Почему нельзя выбирать гиперпараметры по `test`?**
   - Test должен использоваться только один раз в конце для финальной оценки. Если выбирать гиперпараметры по test, информация из test просачивается в процесс обучения, и метрики становятся смещёнными (оптимистичными). Модель перестаёт быть честно оценённой на новых данных.

2. **Почему train-метрика часто, но не всегда выше, чем validation?**
   - Модель обучается на train-данных и может запоминать их шум (переобучаться). Поэтому на train качество часто выше. Однако иногда validation может оказаться выше train из-за случайной вариативности выборки или если train-выборка содержит сложные для модели примеры.

3. **Почему `Pipeline` внутри CV полезен, но не магически гарантирует отсутствие leakage сам по себе?**
   - Pipeline гарантирует, что препроцессинг (стандартизация, кодирование) выполняется внутри каждого фолда CV, а не на всей выборке до разбиения. Это предотвращает утечку информации из валидации в обучение. Однако pipeline не защищает от других видов leakage (например, если признаки созданы с использованием информации из test).

4. **Почему более сложная модель или более долгий тюнинг не всегда побеждают на `test`?**
   - Сложная модель может переобучиться на train-данных и плохо обобщаться. Более долгий тюнинг может подогнать модель под шум валидации, особенно если валидация нестабильна. В итоге на новых (test) данных модель может показать результат хуже простой модели.

5. **Какое didactic shortcut есть в этой ЛР и как вы бы усилили его на продвинутом треке?**
   - В этой ЛР используется один и тот же validation для нескольких решений (выбор feature set и гиперпараметров). На продвинутом треке можно было бы использовать nested CV или отдельный selection split, чтобы сделать оценку ещё более честной и устойчивой к случайности.
